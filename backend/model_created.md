## 1. Главный принцип

Единственное правило, которое определяет всю реализацию (MODEL.md §0, P2):

> **LLM классифицирует и объясняет; детерминированный Python считает каждое число
> и сам вердикт.**

Конкретно:

- **LLM-профайлер** (STEP A) относит бизнес к категориям на 8 осях — возвращает
  только метки из закрытых списков, **ни одного числа**.
- **Код** (STEP B–E) берёт из таблиц коэффициенты и считает выручку, издержки,
  прибыль, точку безубыточности, окупаемость, индексы, score, риск и
  `decision_label` — всё детерминированными формулами, **без ГСЧ и без seed**.
- **LLM** (STEP F) только *формулирует словами* уже посчитанные числа; валидатор
  отклоняет любое выдуманное число и любое несовпадение вердикта.

Каждая «магическая» константа в коде снабжена комментарием `# see MODEL.md §X.Y`
(принцип трассируемости P4 — в коде 52 таких ссылки).

## 3. Конвейер: STEP A → STEP F

Поток данных в `report_agent.py` (MODEL.md §1):

```
INPUT: timeseries (история ~60 мес) + forecast_bundle (Sybilion: forecast+drivers+backtest)
       + descriptions + user_input + forecast_horizon_months
   │
   ▼ STEP A — ПРОФАЙЛЕР (LLM)            §3   → business_profile (8 осей, категории)
   ▼ STEP B — ПАРАМЕТРИЗАЦИЯ (код+таблицы) §4,§5 → коэффициенты {value, source, confidence, corridor}
   ▼ STEP C — ИНДЕКСЫ (код)             §6   → SI, CI, ROI, PB, CONF
   ▼ STEP D — ЭКОНОМИКА (код)           §7   → выручка, прибыль, аналитическая безубыточность, инвестиции
   ▼ STEP E — РЕШЕНИЕ (код)             §8   → score, decision_label, risk_level
   ▼ STEP F — ОБЪЯСНЕНИЕ (LLM+валидатор) §9   → reason
   │
   ▼ REPORT (ARCHITECTURE.md §5.6 + runtime)
```

### STEP A — Профайлер (LLM), §3

Функция `profile_business(descriptions, user_input, *, mode, fallbacks)`.

- Классифицирует бизнес на **8 осях**: `capital_intensity` (5 уровней),
  `perishability`, `seasonality_expectation`, `demand_breadth`, `margin_class`,
  `ticket_class`, `purchase_frequency`, `external_shock_sensitivity`.
- `guard_profile()` — **жёсткий enum-контроль** (§3.4): любое значение вне
  закрытого списка заменяется на документированную «середину»
  (`medium`/`moderate`/`mixed`, и т.д.).
- LLM **не видит** ни истории, ни прогноза, ни денег — только текст описания,
  поэтому физически не может «подогнать» классификацию под результат.
- Поле `confidence` от LLM — самооценка, **в `CONF` не входит** (§3.3, §6.5).
- Профайл можно **инжектировать** через параметр `business_profile=` (используется
  оркестратором, демо и тестами — даёт детерминизм без живого LLM).

### STEP B — Параметризация: категории → коэффициенты, §4, §5

`resolve_parameters(profile, user_params)` + таблицы §5:

- `§5.1` корзина (`ticket_class`): база **6 / 25 / 60 €**, коридоры;
- `§5.2` маржа (`margin_class`): **0.20 / 0.50 / 0.70**;
- `§5.3` профиль окупаемости (`capital_intensity`, 5 уровней): база инвестиций
  **12k/25k/80k/250k/600k**, «нормальная»/«терпимая» окупаемость
  **8·12·24·42·60 / 15·24·42·72·96** мес;
- `§5.4` волатильность спроса **σ** = вклад `demand_breadth` + надбавка
  `perishability` + надбавка `external_shock_sensitivity`, с коридором
  **[0.10, 0.35]**;
- `§5.6` постоянные издержки: аренда **6000** + персонал **9000** + прочее **2400**.

**Каскад разрешения параметра** (§5.7), реализован как `ResolvedParam(value,
source, confidence, corridor)`:

```
1. есть в user_input/overrides  → source="user",    confidence=1.0
2. выводимо из данных           → source="derived",  confidence=0.7
3. иначе — база из таблицы      → source="default",  confidence=0.5
```

Важно (v2.0): **разреженный ввод пользователя НЕ штрафуется** — недостающие
параметры берутся из таблиц, это нормальная работа. Источник параметра
показывается на дашборде ради прозрачности, но на `CONF` не влияет.

### STEP C — Индексы (детерминированные), §6

| Индекс | Функция | Формула / смысл |
|---|---|---|
| **SI** | `seasonality_index` | `(max(s) − min(s)) / mean(s)` — амплитуда сезонности (§6.1) |
| **CI** | `capital_intensity_index` | `I / monthly_revenue` — сколько месяцев выручки «заморожено» (§6.2) |
| **ROI** | `roi_index` | `(monthly_profit × 12) / I` — годовая рентабельность (§6.3) |
| **PB** | `payback_index` | `I / monthly_profit` (или ∞ при прибыли ≤ 0) (§6.4) |
| **CONF** | `confidence_index` | составная уверенность (§6.5) |

**CONF** (§6.5) — прямой ответ на критерий «точность, а не bullshit». Это **не**
один MAPE, а взвешенная смесь трёх активных компонент:

```
c_forecast = clip(1 − mape/0.5, 0, 1)        # точность прогноза Sybilion
c_data     = clip(L/60, 0, 1)                # достаточность истории (5 лет → 1.0)
c_band     = clip(1 − объёмно-взвешенная_ширина_полосы, 0, 1)
c_assumptions = 1.0                          # фиксировано: НЕ штрафует разреженный ввод

CONF = 0.50·c_forecast + 0.25·c_data + 0.25·c_band
```

`c_band` считается **объёмно-взвешенно** (`Σ(high−low)/Σ(forecast)`), чтобы мёртвый
зимний месяц у сезонного бизнеса не раздувал относительную ширину полосы.

### STEP D — Экономическое ядро, §7

**Каскад `scale`** (§7.1) — вместо магической `BASKETS_PER_INDEX_POINT = 23`:

```
1. в описаниях есть оценка трафика  → scale = транзакции/мес / level_idx   (derived)
2. пользователь дал целевую выручку → обратная задача                       (user)
3. иначе — база по ticket_class     → low=30 / medium=12 / high=5           (default)
```

**Выручка/прибыль** (§7.2, §7.3) — `compute_economics(...)`:

- из горизонта прогноза оценивается **дезонализованный уровень спроса**
  `level_idx`, чтобы тройка выручка/издержки/прибыль была внутренне согласована
  (`revenue − costs = profit`, принцип P1) и совпадала по базе с годовой
  безубыточностью;
- `monthly_revenue = level_idx × scale × price`;
- `monthly_profit = monthly_revenue × margin − fixed`.

**Аналитическая безубыточность** (§7.3) — заменяет Monte-Carlo на 20 000 сэмплов:

```
σ_profit = monthly_revenue × margin × σ
break_even = среднее по 12 месяцам от  1 − Φ((fixed − monthly_revenue·s[m]·margin)/σ_profit)
```

Φ — нормальная CDF через `math.erf` (без зависимости от scipy). Детерминировано,
без seed, мгновенно. `downside/upside = monthly_profit ∓ Z_P90 · σ_profit`.

**Инвестиции — двусторонняя оценка** (§7.4) `compute_investment(...)`:

```
I_topdown  = monthly_revenue × CI_target      # «месяцев выручки заморожено»
I_bottomup = база инвестиций профиля (§5.3)    # из статей затрат
I          = (I_topdown + I_bottomup) / 2
divergence = |I_topdown − I_bottomup| / I      # сигнал неопределённости → в риск (§8.4)
```

Разбивка (`breakdown`) распределяется по статьям так, чтобы её сумма **в точности**
равнялась `estimated_initial_investment_eur` (инвариант контракта, проверяется
тестом).

### STEP E — Многомерное решение, §8

`score` (§8.2) — 4 нормированные компоненты:
```
score = 100·(0.35·be + 0.25·pb + 0.20·roi + 0.20·conf)
```

`decision_label` (§8.3) — **адаптивные пороги от профиля**: окупаемость PB
сравнивается с «нормальной»/«терпимой» окупаемостью *именно этого* профиля
(§5.3), а не с универсальным числом:
```
если be≥0.6 И pb_ok И CONF≥0.6 И риск≤medium      → "Launch"
иначе если be≥0.5 И pb_tolerable И CONF≥0.45        → "Adapt concept"
иначе если be≥0.4                                   → "Delay"
иначе                                               → "Do not launch"
# жёсткие override:
если CONF<0.4 → не выше "Delay"          (нет Launch на ненадёжных данных)
если mape>0.4 → не выше "Adapt concept"  (слабый бэктест не даёт зелёный свет)
```

`risk_level` (§8.4) — 4 компоненты:
```
risk = 0.40·σ_norm + 0.30·сезонность_norm + 0.20·ширина_полосы_norm + 0.10·divergence_norm
low (<0.33) / medium (<0.66) / high (иначе)
```

Так вердикт стал **многомерным**: зависит от безубыточности И окупаемости И
уверенности И риска — а не только от безубыточности, как раньше.

### STEP F — Объяснение (LLM над готовыми числами), §9

`build_reason(payload, drivers, *, mode, fallbacks)`:

- LLM получает **только посчитанные числа и категории** (`_reason_payload`, §9.1) —
  ни истории, ни прогноза;
- строгий системный промпт (§9.2): «объясняешь УЖЕ принятое решение, менять числа
  и вердикт ЗАПРЕЩЕНО, использовать ТОЛЬКО числа из входа»;
- `_validate_reason` (§9.3): отклоняет ответ, если в тексте есть число, которого нет
  во входе (с допуском на округление), или если вердикт не совпал;
- при провале — детерминированный шаблон (`_template_reason`) + запись в
  `runtime.fallbacks` (в `prod`), либо ошибка (в `dev`).

Это и есть гарантия «не bullshit»: LLM физически нечего подставить, кроме наших
чисел, а валидатор ловит любое выдуманное.

## 4. Режимы `dev` / `prod` и `runtime.fallbacks`

Управляется переменной окружения `MODEL_MODE` (default **`prod`**), MODEL.md §1.1 /
P6. Реализовано как сквозное правило (`_run_mode()` читает env в момент вызова).

| Точка отказа | `dev` (без фолбэков) | `prod` (фолбэк + громкое сообщение) |
|---|---|---|
| Профайлер-LLM недоступен | **ошибка** | нейтральный профиль + `profiler_unavailable->neutral_profile` |
| Sybilion недоступен (синтетика) | **ошибка** | синтетический прогноз + `sybilion_unavailable->synthetic_forecast` |
| LLM-объяснение не прошло | **ошибка** | шаблон + `llm_explanation_unavailable->template` (или `...invalid->template`) |

Отчёт всегда несёт блок:
```json
"runtime": { "mode": "prod", "fallbacks": ["..."] }
```
Пустой `fallbacks` = всё посчитано на реальных путях. Источник `cache` у Sybilion
**не** считается фолбэком (это ранее-реальные данные); фолбэк — только `mock`
(синтетика).

Зачем так: в `dev` тихий фолбэк опасен — он маскирует баг (думаешь, что работает
реальный путь, а на деле заглушка). В `prod` на сцене нельзя падать, но каждый
фолбэк **громкий** — жюри и пользователь честно видят, что результат частично на
запасном пути.

## 5. Контракт горизонта прогноза

`forecast_horizon_months` (MODEL.md §2.2): default **6**, диапазон **1–12**,
значения вне диапазона клэмпятся (в `dev` — это не ошибка, просто клэмп). Значение:

- передаётся в `sybilion_client.get_forecast(...)` как `soft_horizon`;
- ограничивает число точек прогноза (`forecast[:H]`), что влияет на оценку уровня
  спроса в §7.2 и на графики;
- проходит весь публичный API (`build_report`, `prepare_context`, HTTP-запросы).

Проверка: горизонт 3 и 6 дают разную выручку (разные месяцы формируют уровень), а
длина `graphs.demand_forecast` равна (клэмпнутому) горизонту.

## 6. Публичный интерфейс

Сохранён и расширён горизонтом/профилем (имена не переименованы):

```python
def build_report(
    timeseries: dict[str, float],
    metadata: dict | None = None,
    user_input: str = "",
    descriptions: list[str] | None = None,
    *,
    forecast_horizon_months: int = 6,
    forecast_bundle: dict | None = None,
    business_profile: dict | None = None,   # инъекция профиля (оркестратор/демо/тесты)
) -> dict: ...

def prepare_context(...) -> ReportContext   # один раз тянет прогноз + один раз профайлер
def recompute(context, overrides) -> dict   # what-if: переиспользует кэш, БЕЗ вызова Sybilion и профайлера
```

- `prepare_context` кэширует **и прогноз, и профиль** на контексте; `recompute`
  переигрывает только параметры → экономику → решение → объяснение. Это движок
  живого what-if (`POST /report/recompute`).
- HTTP-слой сохранён: `POST /report/build`, `POST /report/recompute`,
  `GET /health` (теперь возвращает ещё и `mode`). В модели запроса добавлены
  `forecast_horizon_months` и `business_profile`.

**Ключи overrides для what-if:** `monthly_rent_eur`, `staff_costs_eur`,
`other_fixed_eur`, `average_basket_price_eur`, `gross_margin_pct` (или
`gross_margin`), `target_revenue_eur`, `transactions_per_month`.

## 7. Формат отчёта (выход)

Точно по `ARCHITECTURE.md` §5.6 + новый блок `runtime`, валидируется Pydantic
(`_Report`) перед возвратом:

```json
{
  "decision":        { "label", "score", "risk_level", "confidence", "summary" },
  "expected_revenue":{ "expected_monthly_revenue_eur", "..._costs_eur", "..._profit_eur",
                       "downside_monthly_profit_eur", "upside_monthly_profit_eur",
                       "break_even_probability", "payback_months" },
  "investment_cost": { "estimated_initial_investment_eur", "breakdown": [ { "category", "amount" } ] },
  "graphs":          { "demand_forecast": [...], "historical_series": [...] },
  "drivers":         [ { "name", "importance", "direction", "horizon" } ],
  "backtest":        { "mape", "rmse", "quality" },
  "reason":          { "main_reason", "positive_factors", "negative_factors", "recommended_actions" },
  "runtime":         { "mode", "fallbacks" }
}
```

- `decision.label` ∈ **Launch · Adapt concept · Delay · Do not launch**;
- `investment_cost.estimated_initial_investment_eur` == Σ `breakdown[].amount`
  (инвариант, проверяется тестом).

**Эталонный выход для шоукейса** (мороженое в Вене, оффлайн, инжектированный
профиль `ICE_CREAM_PROFILE`):

| Поле | Значение |
|---|---|
| `decision.label` | **Adapt concept** |
| `decision.score` | 63 |
| `decision.risk_level` | **High** |
| `decision.confidence` | 0.84 |
| выручка / издержки / прибыль | €30 405 / €26 522 / €3 884 |
| downside / upside | −€4 845 / €12 612 |
| break-even | 0.65 |
| окупаемость | 18 мес |
| инвестиции | €70 405 (= сумма разбивки) |

Сюжет вердикта: безубыточность (0.65) и окупаемость (18 мес) хорошие, но вердикт
держится на **«Adapt concept»** из-за **высокого риска** (сильная сезонность,
скоропортящийся продукт, зависимость от погоды) — ровно то, что должна показывать
многомерная модель.

## 8. Ключевые инженерные решения и зафиксированные расхождения

Там, где спецификация была недоопределена, выбор **задокументирован в коде**
(а не выдуман молча) — это прямо требовалось заданием:

1. **Безубыточность шоукейса ≈ 0.65, а не 0.55–0.60 из §7.3.** Кэшированный
   прогноз несёт YoY-рост (~+4.7%), поэтому дезонализованный уровень выше
   исторического среднего. Я оставил формулы как в спеке, а в тесте
   `test_baseline_is_adapt_concept` документировал, что **метка не изменилась**
   («Adapt concept») — сместился лишь механизм: с пограничной безубыточности на
   ось риска. Это более сильная демонстрация многомерности.
2. **Интерпретация головной тройки (§7.2 vs §7.3).** «Среднее по горизонту» из
   §7.2 трактовано как оценка *уровня* спроса, который затем дезонализуется — так
   `revenue − costs = profit` (внутренняя согласованность P1) и головные числа
   совпадают по базе с годовой безубыточностью. Задокументировано в
   `compute_economics`.
3. **`CI_TARGET` (множители 1.5/1.7/2.0/3.0/5.0).** §7.4 говорит «переведено в

## 9. Тесты

`tests/test_report_agent.py` — **15 тестов, все зелёные**
(`pytest tests/test_report_agent.py -v`). Старые тесты **мигрированы**, не удалены.

**Мигрированные (сохранён смысл):**
- форма контракта **+ блок `runtime` + инвариант суммы инвестиций**;
- базовый вердикт = «Adapt concept» (теперь многомерный, через ось риска);
- рост аренды → прибыль↓, безубыточность↓, метка → Delay/Do not launch;
- премиум-корзина → улучшение (прибыль/безубыточность/score растут);
- `CONF` вместо `confidence_from_backtest` (выше MAPE → ниже CONF; меньше истории → ниже);
- безубыточность внутри полосы (downside < expected < upside);
- вместо сабпроцесс-codegen — тест **детерминизма/воспроизводимости** (одинаковый
  вход → побайтово одинаковый выход; безубыточность аналитическая, без ГСЧ).

**Новые под v2.0:**
- enum-контроль профайлера + нейтральный фолбэк;
- горизонт соблюдается и клэмпится;
- низкий `CONF` / высокий MAPE **не даёт** «Launch» (override §8.3);
- `runtime.fallbacks` наполняется в `prod`; в `dev` — громкая ошибка;
- валидатор объяснения отклоняет выдуманные числа;
- индекс сезонности отражает амплитуду.

Полный набор Block 4 (`report_agent` + `sybilion_client` + `data_engineer`) —
**26 тестов зелёные**, т.е. правка горизонта в `sybilion_client` ничего не сломала.

> Примечание: в репозитории есть **доранее существующий** сбой сборки
> `tests/test_translation_agent.py` (импорт несуществующего
> `descriptions_to_keywords`) — это дрейф тестов/кода Block 3, я эти файлы не
> трогал и не маскировал.

## 10. Как запустить

```bash
# из корня репозитория

# 1) демо-отчёт по шоукейсу (печатает валидный §5.6 + runtime отчёт)
python -m backend.report_agent

# 2) тесты агента-отчёта
pytest tests/test_report_agent.py -v

# 3) standalone HTTP-сервис (если нужен отдельно)
uvicorn backend.report_agent:app --port 8004
#   POST /report/build      { timeseries, timeseries_metadata, userInput, descriptions,
#                             forecast_horizon_months?, business_profile? }
#   POST /report/recompute  { ..., overrides }
#   GET  /health
```

Переменные окружения:
```bash
MODEL_MODE=prod            # dev | prod (default prod) — см. §4
FEATHERLESS_API_KEY=...    # для профайлера и LLM-объяснения; без ключа в prod → фолбэки
SYBILION_API_TOKEN=...     # для живого прогноза; без токена → кэш/синтетика
```

Без ключей всё работает оффлайн: прогноз из `mock/`, профиль инжектируется
(`ICE_CREAM_PROFILE`) или берётся нейтральный, объяснение — детерминированный
шаблон. Экономика **всегда** детерминирована и воспроизводима.
